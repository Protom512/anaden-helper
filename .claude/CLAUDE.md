# 開発ルール（自律型開発組織プロトコル）

このファイルはClaudeがこのリポジトリで作業するときに常に参照するルールです。
作業状態は `TASKS.md` に永続化し、どの時点で中断・再開しても継続できる設計にする。

---

## /loop 起動・再開プロトコル

`/loop` で作業を開始・再開するとき、**最初に必ず以下を実行する**。

```
1. TASKS.md を Read する
2. ステータスが 🔄 In Progress または 🔍 Reviewing のタスクを探す
3. 見つかった場合 → そのタスクの「再開ポイント」から作業を継続する
4. 見つからない場合 → 📋 Backlog の先頭タスクを取り出して着手する
5. TASKS.md が存在しない場合 → 新規作成してからユーザーに方針確認を求める
```

> **記憶はファイルにある。TASKS.md を読むまで何もするな。**

## TASKS.md の書き方

### ファイル構造

```markdown
# TASKS

## 現在の作業（再開ポイント）

### 🔄 [タスクID] タスク名
- **目的**: （何のためか）
- **再開ポイント**: （次に何をすべきか。具体的なファイル名・関数名・コマンドまで書く）
- **完了条件**: （何をもってDoneとするか）
- **メモ**: （判断の経緯・ハマった点・注意事項）

---

## バックログ

### 📋 [タスクID] タスク名
- **目的**: ...
- **完了条件**: ...

---

## 完了済み

### ✅ [タスクID] タスク名
- **完了日時**: YYYY-MM-DD
- **成果**: （何ができるようになったか）
- **品質チェック**: RELIABILITY✅ PERFORMANCE✅ EXTENSIBILITY✅ GOVERNANCE✅ SECURITY✅ INTEGRATION✅
```

### TASKS.md の更新タイミング（厳守）

| タイミング | 更新内容 |
|-----------|---------|
| タスク着手時 | ステータスを 📋→🔄 に変更。「再開ポイント」に次のアクションを記載 |
| サブタスク完了ごと | 「再開ポイント」を更新（「〇〇ファイルのXX関数を実装済み。次はテスト作成」など） |
| 中断しそうなとき | 「再開ポイント」を最新化してから止まる |
| レビュー開始時 | ステータスを 🔄→🔍 に変更 |
| タスク完了時 | ステータスを ✅ に変更し「完了済み」セクションへ移動 |

> **コードを1つ変更したらTASKS.mdも更新する。セットで動く。**

## 継続ループのワークフロー

### フェーズ0｜起動チェック（毎回必須）

```
Read TASKS.md
→ 中断タスクあり？ → YES: フェーズ2「再開ポイント」から継続
                   → NO:  フェーズ1から開始
```

### フェーズ1｜要件定義と仕様合意

新規タスクに着手する前に以下を提示してユーザーの承認を得る。

- どんな画面・機能になるか（操作の流れを日本語で説明）
- 技術選定の方針
- TASKS.md への起票内容（Backlog に追加するタスク一覧）

> **承認なしに実装を開始しない。**

### フェーズ2｜実装

1. 着手するタスクのステータスを 🔄 に変更し TASKS.md を更新
2. 既存ファイルは変更前に必ず `Read` で確認
3. コードを実装
4. **サブタスクが1つ終わるたびに TASKS.md の「再開ポイント」を更新**
5. テストを書いて `bash` で実行・合格確認
6. README / CHANGELOG を必要に応じて更新

### フェーズ3｜6つの品質レビュー

実装完了後、以下を1項目ずつ自己確認する。NOGOなら自律修正→再テスト。

| # | 観点 | チェック内容 |
|---|------|-------------|
| 1 | **RELIABILITY** | エラーハンドリング・null対応・リソース解放は適切か |
| 2 | **PERFORMANCE** | 不要なループ・N+1・冗長な処理はないか |
| 3 | **EXTENSIBILITY** | 単一責任・命名の明快さ・将来の変更への耐性はあるか |
| 4 | **GOVERNANCE** | コメント・README・CHANGELOGは最新か |
| 5 | **SECURITY** | 入力値のサニタイズ・インジェクション対策・依存関係の脆弱性はないか |
| 6 | **INTEGRATION** | 既存ビルドを壊していないか。テストはすべてパスしているか |

### フェーズ4｜完了処理と次タスクへ

```
1. TASKS.md を更新（タスクを ✅ 完了済みへ移動）
2. ユーザーに完了報告（下記フォーマット）
3. Backlog に次タスクがあれば → フェーズ0 に戻りループ継続
4. Backlog が空なら → ユーザーに次の方針を確認
```

#### 完了報告フォーマット（技術用語禁止）

```
## ✅ 完了：[タスク名]

**できるようになったこと**
（操作・体験の変化を非技術用語で説明）

**品質チェック**
RELIABILITY✅ PERFORMANCE✅ EXTENSIBILITY✅ GOVERNANCE✅ SECURITY✅ INTEGRATION✅

**次のアクション**
- 次タスク: [タスク名]（自動継続）
  または
- Backlog が空です。次に何を作りますか？

**進捗**
| タスク | ステータス |
|--------|-----------|
| xxx    | ✅ Done   |
| yyy    | 🔄 In Progress |
| zzz    | 📋 Backlog |
```

## コーディング規約

### Wiki・ドキュメント
- **GitHub Wiki** は `docs/anaden-helper.wiki/` にある。このディレクトリは独立した git リポジトリ（`anaden-helper.wiki.git`）
- Wiki ページを追加・更新したら、`docs/anaden-helper.wiki/` 内で `git add && git commit && git push` する
- Wiki の各ページは個別の `.md` ファイル（例: `Architecture.md`, `Quick-Start.md`）
- アーキテクチャ変更、新機能追加、重要な設計判断をした場合は Wiki も更新する

### ファイル操作
- 既存ファイルを変更する前に必ず `Read` で内容を確認する
- 複数ファイルにまたがる変更は依存関係を把握してから行う
- 削除はユーザーに確認してから行う

### テスト
- 新機能には必ずテストを追加する
- テストは `bash` で実際に実行してパスを確認してから完了とする
- テストが落ちたらコードを直す（テストを弱くしない）

### コミット・変更管理
- 1コミット = 1つの論理的な変更
- CHANGELOG.md があれば更新する
- breaking change は必ずユーザーに事前報告する

### セキュリティ
- 秘密鍵・トークン・パスワードをコードにハードコードしない
- 環境変数は `.env.example` にキー名だけ記載する
- ユーザー入力は常にバリデーション・サニタイズを行う

## 新規プロダクト立ち上げ時の優先順位

1. **動くものを早く作る** — 完璧より動作確認を優先
2. **スケールより明快さ** — 最初から過剰な抽象化をしない
3. **依存関係は最小限に** — 後から足すほうが引くより簡単
4. **ドキュメントは実装と同時に** — 後回しにしない

---

## プロセス改善ルール

同じ種類のバグやレビュー差し戻しが2回以上発生した場合：

1. TASKS.md に「プロセス改善」タスクを Backlog に起票する
2. CLAUDE.md へのルール追加をユーザーに提案する
3. 承認されたらこのファイルを更新する

## ステータス凡例

| 記号 | 意味 |
|------|------|
| 📋 Backlog | 着手待ち |
| 🔄 In Progress | 実装中（再開ポイント必須） |
| 🔍 Reviewing | セルフレビュー中 |
| ✅ Done | 完了・テストパス済み |
| ⚠️ Blocked | 外部待ち・判断待ち（理由を必ず記載） |
