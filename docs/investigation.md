## 1. Tiny Vision LLM アプローチ

### 1-1. モチベーション

現状のテンプレートマッチングは実用的だが、以下の限界がある：

| 限界 | 具体例 |
|------|--------|
| 輝度変化に弱い | 自動輝度調整、ナイトモードでマッチ精度が落ちる |
| 新規画面の対応コスト | 新しいミニゲームが出るたびにテンプレートを手動追加 |
| アニメーション中の不定性 | UI の出現・消滅タイミングで Unknown になりがち |
| 解像度依存 | Pixel 7a 以外の端末でテンプレートが使えない |

→ **超小型ビジョンモデル（CNN 分類器）** で「画面の状態」を直接分類する。

### 1-2. 候補モデル

| モデル | パラメータ数 | 推論レイテンシ（CPU） | 特徴 |
|--------|-------------|---------------------|------|
| **カスタム小型 CNN** | **0.1M〜0.4M** | **<5ms** | ゲーム特化、最軽量 |
| **MobileNetV3-Small** | **~2.5M** | ~5ms | ImageNet 事前学習済み |
| **EfficientNet-Lite0** | **~4.7M** | ~12ms | 高精度 |
| **SqueezeNet 1.1** | **~1.2M** | ~8ms | 超軽量レガシー |

**推奨**: まずカスタム小型 CNN（~40万パラメータ）。アナザーエデンの状態は 15〜20 クラスしかなく、ImageNet のような 1000 クラス分類と比べて極めて単純。

```
カスタム CNN アーキテクチャ案:
Input (224x224x3)
  → Conv2D(32, 3x3, stride=2) + BN + ReLU    # 112x112x32
  → DepthwiseConv(32, 3x3) + BN + ReLU
  → Conv2D(64, 1x1) + BN + ReLU
  → MaxPool(2x2)                               # 56x56x64
  → DepthwiseConv(64, 3x3) + BN + ReLU
  → Conv2D(128, 1x1) + BN + ReLU
  → GlobalAvgPool → Dense(64) → Dense(15) + Softmax
```

### 1-3. Rust での推論

| フレームワーク | 特徴 | 推奨度 |
|---------------|------|--------|
| **ONNX Runtime (`ort` crate)** | C++ バックエンド、高速、成熟 | ⭐⭐⭐ |
| **Candle** (Hugging Face) | 純 Rust、Python 依存なし | ⭐⭐ |
| **tract** | 純 Rust、ONNX 対応 | ⭐⭐ |

**実用レイテンシ**: 全候補で **2〜15ms**。200ms 要件を余裕でクリア。

PyTorch で学習 → ONNX エクスポート → `ort` crate で Rust 推論、が最も安定。

### 1-4. 学習データ要件

| アプローチ | 1クラスあたりの枚数 | 全クラス（15）合計 |
|-----------|-------------------|-------------------|
| ゼロから学習 | 500〜1,000枚 | 7,500〜15,000枚 |
| 転移学習（MobileNetV3） | 100〜300枚 | 1,500〜4,500枚 |
| Few-shot + 強 Augmentation | 50〜100枚 | 750〜1,500枚 |

ゲーム画面に有効な Data Augmentation:
- ✅ 輝度・コントラスト変動（±15%）: 自動輝度調整対策
- ✅ 色相シフト（±5%）: ディスプレイ設定差異対策
- ✅ クロップ（中心 90%）: キャプチャ枠のズレ対策
- ❌ 回転・反転（ゲーム画面は常に正面）

### 1-5. テンプレートマッチング vs ディープラーニング

| 観点 | テンプレートマッチング | 小型 CNN |
|------|---------------------|---------|
| 堅牢性 | 低（輝度・解像度に弱い） | 高（汎化能力あり） |
| 推論速度 | 10〜50ms（ダウンスケールあり） | 2〜15ms |
| 新規対応コスト | テンプレート追加（簡単） | データ追加 + 再学習 |
| デバッグ性 | 高（マッチ箇所が視覚的） | 低（ブラックボックス） |
| 実装複雑さ | 低（既に動作） | 中（学習パイプライン必要） |

**推奨**: ハイブリッド 2 層構造

```
第1層: CNN → トップレベル分類（Title/Home/Battle/Menu/Dialog/Loading）
第2層: テンプレートマッチング → 詳細分類（BattlePhase, MenuTab 等）
```

## 2. YouTube プレイ動画からの学習データ活用

### 2-1. 動画ダウンロード

```bash
# yt-dlp で 720p 動画をダウンロード
yt-dlp -f "bestvideo[height<=720]" -o "data/raw/%(title)s_%(id)s.mp4" <URL>

# プレイリスト一括
yt-dlp -f "bestvideo[height<=720]" --yes-playlist -o "data/raw/%(playlist_index)03d.mp4" <PLAYLIST>

# 特定時間範囲
yt-dlp --download-sections "*0:05:00-0:30:00" -o "data/raw/section.mp4" <URL>
```

720p で十分。ゲーム UI はこの解像度で明瞭に認識可能。

### 2-2. フレーム抽出戦略

**推奨: シーン変化検出 + 定間隔サンプリングのハイブリッド**

```bash
# 定間隔（1fps）
ffmpeg -i input.mp4 -vf "fps=1" "frames/%04d.jpg"

# シーン変化検出
ffmpeg -i input.mp4 -filter:v "select='gt(scene,0.3)'" -vsync vfr "scenes/%04d.jpg"
```

| 方法 | 10時間動画からの枚数 | 長所 | 短所 |
|------|-------------------|------|------|
| 定間隔 1fps | 36,000枚 | 簡単 | 重複多い |
| シーン変化検出 | 1,000〜5,000枚 | 重複少ない | 閾値チューニング必要 |
| **ハイブリッド** | **3,000〜8,000枚** | **バランス良い** | やや複雑 |

### 2-3. 半自動ラベリング（現プロジェクトの強み）

**最大の利点**: 既存の `TemplateMatcher` + `SceneDetector` が動作している。

```
パイプライン:
1. 抽出フレームに SceneDetector::detect_state() を実行
2. confidence > 0.85 → 「高信頼ラベル」として自動採用
3. Unknown / 低信頼度 → 「要手動ラベリング」としてマーク

自動ラベリング率の想定:
- TitleScreen, Loading: 95%以上 自動可能
- HomeScreen, InBattle: 80〜90% 自動可能
- Menu, Dialog: 60〜80% 自動可能
- MiniGame: テンプレート次第
```

→ **手動ラベリングは 500〜1,000枚程度**で済む（全 5,000〜10,000枚のうち）。

### 2-4. 法的・利用規約の考察

| 項目 | リスク | 対策 |
|------|--------|------|
| YouTube ToS（ダウンロード禁止） | 中 | 個人利用・研究目的に限定 |
| ゲーム画面の著作権 | 低 | キャプチャは一般的にフェアユース |
| 配信者の録画・編集物 | 中 | 画面のみ抽出、音声は使わない |
| データの再配布 | 高 | 学習データは非公開、コミットしない |

**推奨プラクティス:**
- 音声は一切使用しない（フレーム画像のみ）
- 可能であれば **自分でプレイしたキャプチャを優先**
- YouTube 動画は「特定の稀な画面状態」の補完用途に留める

### 2-5. 必要な動画時間

| 状態カテゴリ | 必要な動画時間 | 備考 |
|------------|--------------|------|
| TitleScreen | 5分 | 起動シーンを含む動画 |
| HomeScreen | 1〜2時間 | ほぼ全動画に含まれる |
| InBattle（各 Phase） | 3〜5時間 | 戦闘動画が最も多い |
| Menu（各 Tab） | 2〜3時間 | 育成・編成動画 |
| Dialog（各種） | 2〜3時間 | ストーリー動画 |
| Loading | 1〜2時間 | 遷移の合間に多数 |
| MiniGame（釣り等） | 1〜2時間 | 釣り特化動画 |
| **合計** | **10〜15時間** | |

1fps 抽出 → 約 36,000 フレーム → 重複排除後 **約 3,000〜5,000枚**。

## 3. 統合ロードマップ

### Phase 1: テンプレートマッチングの強化（今すぐ・コストゼロ）

```
現状のままテンプレートを追加し、カバレッジを上げる。
各画面のテンプレートを 3〜5 個ずつ用意する。
```

### Phase 2: カスタム CNN の導入（1〜2週間）

```
1. 自分で 1〜2時間プレイ → ADB 連続キャプチャ（0.5fps）
2. 約 2,000〜4,000枚のキャプチャを取得
3. 既存 SceneDetector で自動ラベリング（約 70%）
4. 残りを手動ラベリング（約 600〜1,200枚）
5. カスタム CNN を PyTorch で学習 → ONNX エクスポート
6. Rust (ort crate) で推論組み込み
```

### Phase 3: YouTube データでレアケース補完（必要に応じて）

```
1. yt-dlp で 10〜15時間分のプレイ動画をダウンロード
2. ハイブリッド抽出で約 5,000〜10,000枚
3. Phase 2 の CNN で事前推論 → 不確実なフレームのみ手動ラベリング
4. 全データで再学習 → 精度向上
```

### Phase 4: ハイブリッド推論への移行

```
anaden-vision の SceneDetector を差し替え:

第1層: CNN → トップレベル分類（速い・堅牢）
第2層: テンプレートマッチング → 詳細分類（確実・解釈可能）

VisionEngine trait で両方を透過的に切り替え可能にする。
```

## アーキテクチャへの影響

現在の設計は `anaden-vision` に画像認識をカプセル化しているため、
CNN 導入時も **影響範囲は `anaden-vision` 内部のみ**。

```rust
// 将来の VisionEngine trait（anaden-vision 内部）
trait VisionEngine: Send + Sync {
    fn detect_scene(&self, screenshot: &DynamicImage) -> RecognitionResult;
}

struct TemplateVisionEngine { /* 既存 */ }
struct CnnVisionEngine { /* ONNX モデル */ }
struct HybridVisionEngine { /* CNN + Template */ }
```

上位層（`anaden-engine`, `anaden-cli`）は変更不要。
